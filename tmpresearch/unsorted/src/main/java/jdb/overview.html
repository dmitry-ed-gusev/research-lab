Библиотека классов JDB 10.2.0 (31.05.2011). Библиотека содержит классы для работы с различными типами CУБД - Informix, MySQL, ODBC-источники данных, DBase (dbf), MS SQL Server. Также библиотека содержит различные абстрактные модели баз данных и классы, которые позволяют генерировать эти модели.

Для использования данной библиотеки необходимы следующие библиотеки (зависимости от внешних библиотек):

  • log4j (1.2.15) - модуль ведения журнала работы/ошибок (логгирование).
  • commons-lang (2.4) - модуль, содержащий большой набор общеупотребительных утилитарных функций.
  • commons-configuration (1.6) - модуль, позволяющий хранить параметры во внешнем файле и работать с этим файлом.
  • commons-collections (3.2) - модуль для работы с различными коллекциями данных.
  • commons-logging (1.1) - интерфейсный модуль для многих систем логгирования (журналирования).
  • jlib (4.4.8 и выше) - модуль, содержащий некоторые утилитарные функции (см. документацию).
  • JDBC-драйвера к поддерживаемым типам СУБД (см. ниже).
  • JUnit (4.8.2+) - фреймворк для тестирования исходного кода. Для сборки или использования библиотеки фреймворк не нужен.
Для успешной компиляции исходного кода данной библиотеки нужны не все перечисленные модули (необходимы только модули log4j, commons-lang, commons-configuration, jlib), но для обеспечения полного функционирования всех классов данной библиотеки все эти модули необходимы.

Библиотека поддерживает следующие типы СУБД (для работы необходим соответствующий JDBC-драйвер):

  • Informix (тестировалось с версией СУБД IDS 9.4). Тип соединения DBType.INFORMIX.
  • Mysql (тестировалось с версией 5.0.51). Тип соединения DBType.MYSQL.
  • ODBC-источник данных (с помощью встроенного JAVA-драйвера моста ODBC-JDBC). . Тип соединения DBType.ODBC.
  • DBF (тестировалось с версией dBaseIII и dBaseIV). Тип соединения DBType.DBF.
  • MS SQL (тестировалось с версией MS SQL Server 2005). Тип соединения DBType.MSSQL_NATIVE или DBType.MSSQL_JTDS. Для данного типа СУБД возможно использование двух типов драйверов: драйвера jTDS (open source) и драйвера, поставляемые компанией Microsoft. Рекомендуется использование драйверов jTDS ввиду их большей производительности и стабильности.

Использование логгирования.

Для работы системы логгирования следует инициализировать логгер с именем "jdb" - тогда все классы библиотеки смогут вести журналирование. Если же логгер инициализирован не будет, то при работе библиотеки будет возникать сообщение об этом (WARNING). Также следует инициализировать логгер с именем org.apache.commons для того, чтобы используемые библиотеки commons-lang и commons-configuration тоже могли вести журналирование.

Особенности (фичи) в работе библиотеки.

  • При соединении с СУБД с помощью конфигурации DBConfig, которая умеет харнить списки "разрешенных" и "запрещенных" таблиц, будьте внимательны - списки должны содержать только имена таблиц, без указания схемы или владельца таблицы (например, не рекомендуется хранить таблицы в этих списках в таком виде: dbo.table1, лучше хранить именно имя - table1. Несоблюдение данного правила может повлиять на правильность обработки таблиц в модулях обработки данных (пакет jdb.processing). Также при использовании модулей обработки данных (загрузки/выгрузки, проверки целостности и т.п.) в СУБД, с которыми будут работать эти модули нельзя создавать таблицы с одинаковыми именами в разных схемах данных - это может привести к неоднозначности обработки данных, проблемам с переносом данных из одной СУБД в другую (другого типа), также возможно повреждение данных.
  • Некоторые функции/методы обработки данных не работают (программно заглушены) или работают не очевидным образом для СУБД типа ODBC-источник данных и DBF. Внимательно читайте документацию к используемым модулям и методам!
  • Операции загрузки данных в БД с диска или выгрузки данных на диск из БД работают с использованием несоставного ключевого поля. Если ключевое поле отсутствует или не указано - выполнение операций будет невозможно (возникновение ИС).

8.3.0 (15.06.2010)

Например, если указано выполнение в X потоков, а sql-батч состоит из X*RATIO запросов или более, то будет выполнена многопотоковая обработка, если же батч состоит из меньшего количества запросов, то будет выполнена однопотоковая обработка sql-батча. Это внутренний механизм работы модуля, который может быть изменен в последующих версиях.
